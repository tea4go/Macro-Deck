# 自动化测试

本仓库（Macro Deck 主程序）的自动化测试分两层：

1. **单元测试**（NUnit）——测纯逻辑，随 CI 的 `dotnet test` 运行，无需 GUI。
2. **端到端冒烟测试**（PowerShell）——启动真实 Macro Deck，验证 HTTP 服务、WebSocket、启动日志。需真实 GUI 环境，本机运行。

---

## 1. 单元测试（NUnit）

测试项目位于 `tests/MacroDeck.Tests/`，框架为 NUnit（`[TestFixture]` / `[TestCase]` / `Assert.That`）。

### 运行

```bash
dotnet test tests/MacroDeck.Tests/MacroDeck.Tests.csproj
```

需要 .NET 10 SDK。CI 通过 `.github/workflows/tests.yml` 自动执行同一命令。

### 现有覆盖

| 测试文件 | 覆盖内容 |
|----------|----------|
| `ConvertNameStringTests.cs` | 变量名规范化（小写、分隔符替换、变音字符转写） |
| `StringCipherTest.cs` | StringCipher 加解密往返 |
| `TemplateManagerTests.cs` | Cottle 模板关键字与渲染 |
| `ConditionActionTests.cs` | 条件动作配置序列化（见下） |

### ConditionAction 回归测试

`ConditionActionTests.cs` 是为修复一个真实 bug 而加的回归保护：

> `ConditionAction.UpdateConfiguration()` 曾把 if 分支动作（`_actions`）同时序列化进
> `actions` 和 `actionsElse` 两个配置键，导致在 GUI 里编辑条件动作时，**else 分支被 if 分支覆盖**。

三个测试：

- `UpdateConfiguration_KeepsElseBranchSeparateFromIfBranch` —— if 分支为空、else 分支含 1 项时，断言序列化后 `actionsElse` 仍为 1 项（修复前会被覆盖成 0）。
- `RoundTrip_KeepsBothBranchesIndependent` —— 用 `TypeNameHandling.Auto`（与加载 profile 相同的方式）序列化再反序列化，断言两个分支的动作数量各自正确。
- `RoundTrip_PreservesConditionFields` —— 条件类型/方法/比较值能正确往返。

这些测试只走 JSON 序列化路径，不调用 `Trigger()`，因此不依赖 GUI 或 VariableManager。

## 2. 端到端冒烟测试（test-macrodeck.ps1）

脚本 `test-macrodeck.ps1`（仓库根目录）启动真实 Macro Deck，验证主程序对外服务是否正常。

### 运行

```powershell
# 重启 Macro Deck 后测试
powershell -ExecutionPolicy Bypass -File .\test-macrodeck.ps1

# 不重启，测当前运行的实例
powershell -ExecutionPolicy Bypass -File .\test-macrodeck.ps1 -SkipRestart
```

退出码：`0`=全部 PASS，`1`=有 FAIL（可用于 CI 判定）。

### 参数

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `-SkipRestart` | 关 | 跳过重启，测当前运行实例 |
| `-ClientId` | `lvgoejr` | WebSocket 握手用的已配对客户端 ID（须在 devices.json，Blocked=false） |
| `-ServerHost` | `localhost` | 主机 |
| `-Port` | `8191` | HTTP/WebSocket 端口 |

### 验证项

| 检查项 | 含义 | 判据 |
|--------|------|------|
| `PortReady` | 服务端口已起 | TCP 能连上 8191 |
| `PingEndpoint` | REST 健康检查 | `GET /ping` → 200 |
| `WebClient` | web client 已部署 | `GET /client/` → 200 |
| `WebSocketHandshake` | 客户端协议正常 | `CONNECTED` + `GET_BUTTONS` 收到响应 |
| `StartupLog` | 启动干净 | 日志含 `Loading plugins`、`startup finished`，无 `Failed to start server` |

### 故障排查

**`WebClient FAIL`（/client/ 返回 404）**
Macro Deck 用启动时的工作目录作为 content root，去 `<工作目录>\wwwroot` 找 web client 静态文件。若从错误目录启动（如从命令行当前目录），会找不到 wwwroot 导致 404。脚本已用 `Start-Process -WorkingDirectory <安装目录>` 修正，使其与正常启动一致。若仍 404，确认 `C:\Program Files\Macro Deck\wwwroot\client` 是否存在。

**`WebSocketHandshake FAIL`**
- 客户端 ID 未配对或被封锁：检查 `%APPDATA%\Macro Deck\devices.json`，用 `-ClientId` 指定一个 `Blocked:false` 的已知客户端。
- API 版本过低：握手用的 `API` 值需不低于服务端要求（脚本内为 `"20"`）。

**`StartupLog FAIL`**
查看 `%APPDATA%\Macro Deck\logs\log<日期>.txt`，关注 `Failed to start server`（端口被占用 / SSL 配置等）。

## 3. 已知约束

- **本机无 .NET SDK 时无法跑单元测试**：`dotnet test` 需要 .NET 10 SDK。仅有 runtime 的机器跑不了，需在带 SDK 的环境或 CI 上执行。
- **fork 仓库的 GitHub Actions 需手动启用一次**：fork 默认不自动运行 workflow。首次需到仓库 **Actions** 标签页点 "I understand my workflows, go ahead and enable them"，之后 push 才会触发 CI（`ci.yml` 已配置为 push 到任意分支均触发）。
- **端到端测试需要真实 GUI 环境**：要启动 Macro Deck 桌面程序与已配对客户端，不适合在无头云端 Runner 上运行，适合装有 Macro Deck 的本机或自托管 Runner。

---

## 4. 接入 CI

单元测试已随 `tests.yml`（被 `ci.yml` 调用）自动执行。端到端脚本可在自托管 Runner 上这样判定：

```powershell
powershell -ExecutionPolicy Bypass -File .\test-macrodeck.ps1
if ($LASTEXITCODE -ne 0) { throw "Macro Deck 端到端冒烟测试失败" }
```
